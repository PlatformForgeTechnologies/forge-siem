Overview

Forge SIEM is a security information and event management platform built around one principle: your raw logs never leave your environment. Forge runs detection rules in memory and stores only alert metadata — not the underlying log data.

There are two ways to use Forge:

  • SaaS (BYOC) — You connect your servers and cloud accounts. Forge runs the control plane (rules engine, dashboard, alerting). Your raw logs stay in your environment. This is the recommended path for most teams.
  • Self-hosted — You deploy the full Forge platform into your own Kubernetes cluster using the Helm chart. You operate everything yourself.
Early access: Forge is currently in private early access. The SaaS platform is under active development. Join the waitlist at platformforgetechnologies.com to be notified when access opens.

How BYOC works

In the SaaS (BYOC) model, there are two distinct planes:

  • Forge control plane — Hosted by Forge. Runs the detection rules engine, alert deduplication, dashboard, API, and notification dispatch. You access this at app.platformforgetechnologies.com.
  • Your data plane — Runs in your environment. Consists of either the Forge agent (on servers) or cloud-native collectors (on AWS/Azure/GCP). Raw logs are processed transiently — rules fire in memory, raw data is never forwarded to Forge.

What Forge stores about you: alert metadata (rule name, severity, timestamp, affected host), agent heartbeats, your detection rules, and user accounts. Raw log content is never stored or transmitted to Forge infrastructure.

What you install: Only the agent binary or a DaemonSet — not the full Helm chart. The Helm chart is Forge's own infrastructure, not yours.

Linux / On-prem servers

The Forge agent is a single Go binary that runs as a systemd service. It collects logs, monitors file integrity, tracks running processes, and ships events over mTLS to the Forge ingest endpoint.

1. Generate an enrollment token

In the Forge dashboard, go to Agents → Enrollment Tokens → New Token. Set an expiry and max-use count, then copy the token.

2. Run the install script

shell — Linux
curl -fsSL https://github.com/PlatformForgeTechnologies/\
forge-siem-agent/releases/latest/download/install.sh \
  | ENROLL_TOKEN=<your-token> sh

That's all you need to provide. The script installs the agent binary, exchanges the token for a signed mTLS certificate (handled automatically — no certificate knowledge required), writes config to /etc/forge-siem/, and starts the forge-siem-agent systemd service. The agent appears in your dashboard within 30 seconds.

Kubernetes clusters

For Kubernetes environments, the agent runs as a DaemonSet — one pod per node. Like the Linux install, you only need an enrollment token. The agent handles its own certificate exchange automatically on first run.

1. Create an enrollment token

In the Forge dashboard, go to Agents → Enrollment Tokens → New Token. Copy the token — it's the only credential you need.

2. Create the namespace and token secret

shell
kubectl create namespace siem
kubectl create secret generic forge-enroll-token \
  --namespace siem \
  --from-literal=token=<your-enrollment-token>

3. Deploy the agent DaemonSet

shell
kubectl apply -f https://github.com/PlatformForgeTechnologies/\
forge-siem-agent/releases/latest/download/k8s/daemonset.yaml

On first start, each pod reads the token from the secret, calls POST /enroll on the Forge API, receives its signed mTLS certificate, and immediately begins sending events. The certificate is persisted to the node's filesystem — subsequent restarts skip enrollment and go straight to connecting.

Each agent pod mounts /var/log, /proc, and the host root filesystem. It uses hostNetwork and hostPID so that active response actions apply to the host, not the container namespace.

Helm chart: A Helm chart (forge-siem-agent) is also available at k8s/siem-agent/ in the agent repo for teams that manage deployments via GitOps. The chart requires pre-provisioned TLS secrets — use the DaemonSet path above for the simpler token-only flow.

Network requirements

The Forge agent makes outbound-only connections. No inbound ports need to be opened. Every node running the agent needs egress access to the following:

DestinationProtocol / PortPurpose
api.platformforgetechnologies.comHTTPS / TCP 443Enrollment (one-time) and heartbeat API
ingest.platformforgetechnologies.commTLS TCP / 1514Persistent event stream to Forge ingest
Port 1514 is not HTTPS. Most corporate firewalls and cloud security groups block outbound TCP on non-standard ports by default. You will need an explicit egress rule allowing TCP 1514 to ingest.platformforgetechnologies.com from each agent host or node.

AWS security group example

AWS — outbound rule (console or Terraform)
# Allow agents to reach Forge ingest
Type:        Custom TCP
Port range:  1514
Destination: ingest.platformforgetechnologies.com  (or its resolved IP range)
Description: Forge SIEM ingest (mTLS)

# Allow enrollment + API (likely already open)
Type:        HTTPS
Port range:  443
Destination: 0.0.0.0/0

iptables / nftables example

shell — on each host (or via Ansible)
iptables -A OUTPUT -p tcp --dport 1514 \
  -d ingest.platformforgetechnologies.com -j ACCEPT
iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT

Agent enrollment

Enrollment is a one-time process. The agent sends a CSR to POST /enroll on the Forge API, authenticated by a short-lived enrollment token. The API signs the CSR with the platform CA and returns a client certificate. The agent saves the certificate to /etc/forge-siem/agent.crt and uses it for all subsequent mTLS connections.

Enrollment tokens are scoped by max-use count and expiry. A token cannot be reused after expiry or after reaching its use limit — generate a new token for each batch of enrollments.

Revoking an agent

In the Forge dashboard, go to Agents, find the agent, and click Revoke. The agent's certificate is added to the CRL immediately. The agent will fail its next heartbeat check and stop sending events within 60 seconds.

Agent configuration

After enrollment, the agent writes its config to /etc/forge-siem/config.yaml. Key options:

KeyDefaultDescription
ingest_host(set during enrollment)Forge ingest endpoint — do not change manually
log_paths/var/log/**/*.logGlob patterns for log files to tail
fim.paths/etc, /usr/bin, /usr/sbinDirectories monitored for file integrity changes
fim.interval30sHow often FIM baseline is re-checked
process.enabledtrueWhether to collect process creation events via /proc
heartbeat.interval30sAgent liveness heartbeat to the Forge API

What the agent collects

CategorySourceDescription
Log eventsFile tailing (glob)Auth logs, syslog, application logs, audit.log
File integrity/proc + inotifyHash changes, permission changes, new/deleted files in monitored paths
Process activity/proc scanNew process creation, command-line args, parent PID
HeartbeatAgent internalAgent version, uptime, hostname — no log data

All events are sent as JSON over a persistent mTLS TCP connection (port 1514). Events are buffered locally if the connection is interrupted and replayed on reconnect.

Self-hosted platform

If you want to run the entire Forge SIEM platform yourself — ingest, detection, storage, dashboard — you can deploy it into your own Kubernetes cluster using the Helm chart. This is the right path for air-gapped environments, strict data sovereignty requirements, or development/testing.

Self-hosted vs SaaS: In self-hosted mode, you operate everything: the control plane, the ingest pipeline, all storage backends, and the agent ingest endpoint. This requires significantly more infrastructure than the SaaS agent-only install.

Dev / single-node install

The values-dev.yaml overlay runs all dependencies in-cluster with conservative resource limits — suitable for any single-node dev or test cluster.

shell
helm repo add forge-siem \
  https://platformforgetechnologies.github.io/forge-siem-platform
helm repo update

helm install siem forge-siem/siem \
  --namespace siem --create-namespace \
  -f https://raw.githubusercontent.com/PlatformForgeTechnologies/\
forge-siem-platform/main/k8s-apps/siem/values-dev.yaml

The bootstrap job runs first (PreSync hook), creates the PostgreSQL schema and ClickHouse tables, then the services start. Expect 2–4 minutes on first install. Access the dashboard:

shell
kubectl port-forward -n siem svc/siem-dashboard 8080:80
# Open http://localhost:8080
# Default login: admin@forge-siem.local / ForgeSIEM-dev-admin-2026
Warning: The static passwords in values-dev.yaml are public. Change secrets.* and postgres.staticPassword before exposing any port to the internet.

Production install (BYO dependencies)

For production, use values.yaml with your own managed PostgreSQL, Redis, ClickHouse, and RabbitMQ. Forge SIEM uses External Secrets Operator to pull credentials from AWS Secrets Manager by default.

values.yaml — key overrides
dependencies:
  postgres:   { mode: byo }
  redis:      { mode: byo }
  clickhouse: { mode: byo }
  rabbitmq:   { mode: self_hosted }   # or byo

global:
  secrets:
    createExternalSecrets: true
    externalSecretStoreRef:
      kind: ClusterSecretStore
      name: aws-secrets-manager
  secretsManagerPath: myorg/forge-siem/secrets

ArgoCD setup

The Helm chart includes ArgoCD sync wave annotations. The bootstrap job runs at wave 1 (after wave 0 databases), services run at wave 2.

argocd — Application manifest
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: forge-siem
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/PlatformForgeTechnologies/forge-siem-platform
    targetRevision: main
    path: k8s-apps/siem
    helm:
      valueFiles: [values-dev.yaml]
  destination:
    server: https://kubernetes.default.svc
    namespace: siem
  syncPolicy:
    automated: { prune: true, selfHeal: true }
    syncOptions: [CreateNamespace=true]

Scaling with KEDA

When keda.enabled: true, KEDA ScaledObjects are created for decoder and rules-engine. They scale on Redis stream consumer lag. Minimum replicas: 1. Maximum: 10 (decoder), 5 (rules-engine). Adjust via decoder.keda.* in values.

Upgrades

shell
helm repo update forge-siem
helm upgrade siem forge-siem/siem --namespace siem -f my-values.yaml

The bootstrap job re-runs on every upgrade via the PreSync hook, applying schema migrations idempotently. No manual DB migration steps are needed.
