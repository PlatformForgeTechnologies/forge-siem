Quickstart

Forge SIEM is a Kubernetes-native SIEM + EDR platform that ships as a Helm chart. This guide gets you from zero to your first alert in under 10 minutes on any cluster.

Note: These instructions assume a working kubectl + Helm 3 setup. For local development, k3s or kind work well.

1. Add the Helm repo

shell
helm repo add forge-siem \
  https://platformforgetechnologies.github.io/forge-siem-platform
helm repo update

2. Install

For a Raspberry Pi 5 or any single-node cluster, use the bundled Pi overlay — it runs all dependencies in-cluster with conservative resource limits:

shell
helm install siem forge-siem/siem \
  --namespace siem --create-namespace \
  -f https://raw.githubusercontent.com/PlatformForgeTechnologies/\
forge-siem-platform/main/k8s-apps/siem/values-pi.yaml

3. Wait for pods to become ready

shell
kubectl get pods -n siem -w

The bootstrap job runs first (PreSync hook), creates the PostgreSQL schema and ClickHouse tables, then the services start. Expect 2–4 minutes on first install.

4. Access the dashboard

shell — local port-forward
kubectl port-forward -n siem svc/siem-dashboard 8080:80
# Open http://localhost:8080

Default credentials from values-pi.yaml: admin@forge-siem.local / ForgeSIEM-pi-admin-2026. Change these before exposing the cluster externally.

Pi / single-node install

The values-pi.yaml overlay is the easiest way to run everything on constrained hardware. It sets:

  • All dependencies (postgres, redis, clickhouse, rabbitmq) to mode: self_hosted — deployed in-cluster
  • Single replicas for all services
  • Conservative ClickHouse memory limits (4Gi max)
  • KEDA and serviceMonitor disabled
  • Static secrets (change before exposing publicly)
Warning: The static passwords in values-pi.yaml are public. Change secrets.* and postgres.staticPassword before exposing any port to the internet.

Customising Pi values

Download values-pi.yaml locally and override what you need:

shell
curl -fsSL https://raw.githubusercontent.com/PlatformForgeTechnologies/\
forge-siem-platform/main/k8s-apps/siem/values-pi.yaml > my-values.yaml

# Edit passwords, hostnames, resource limits
vim my-values.yaml

helm install siem forge-siem/siem \
  --namespace siem --create-namespace \
  -f my-values.yaml

Production install (BYO dependencies)

For production, use values.yaml with your own managed PostgreSQL, Redis, ClickHouse, and RabbitMQ. Forge SIEM uses External Secrets Operator to pull credentials from AWS Secrets Manager by default.

values.yaml — key overrides
dependencies:
  postgres:   { mode: byo }
  redis:      { mode: byo }
  clickhouse: { mode: byo }
  rabbitmq:   { mode: self_hosted }   # or byo

global:
  secrets:
    createExternalSecrets: true
    externalSecretStoreRef:
      kind: ClusterSecretStore
      name: aws-secrets-manager
  secretsManagerPath: myorg/forge-siem/secrets

The secrets manager path must contain keys matching the secret key references in global.secretNames.*. See the full values reference below.

Agent enrollment

Agents connect to the ingest service over mTLS. Before an agent can connect, it must enroll via the API to obtain a signed client certificate.

1. Create an enrollment token (admin only)

shell
curl -s -X POST https://siem.example.com/api/v1/enrollment/tokens \
  -H "Cookie: $SESSION_COOKIE" \
  -H "Content-Type: application/json" \
  -d '{"max_uses": 5, "ttl_hours": 24}' | jq .token

2. Install and enroll the agent

shell — Linux / macOS
curl -fsSL https://github.com/PlatformForgeTechnologies/\
forge-siem-agent/releases/latest/download/install.sh \
  | INGEST_HOST=siem-ingest.siem.svc.cluster.local \
    ENROLL_TOKEN=<token-from-step-1> sh

The agent will enroll via POST /enroll, receive a signed client certificate from the CA, save it to /etc/forge-siem/agent.crt, and start sending events immediately.

Email / SMTP

SMTP is configured in the Admin Center under Platform Settings. The platform uses SMTP for alert notifications and (in future) user account emails.

Tested with Brevo (formerly Sendinblue) on port 2525 (alternative SMTP — Linode blocks 587). Ensure your sender domain is verified in your SMTP provider's dashboard before testing.

Redis streams

All inter-service communication uses Redis Streams with consumer groups. The stream topology is:

topology
siem:raw-events      → decoder, raw-archiver
siem:decoded-events  → rules-engine
siem:alerts          → alert-dedup, alert-indexer, compliance-tagger
siem:deduped-alerts  → response-orchestrator, notification-dispatcher

Each consumer uses XReadGroup + XAutoClaim for at-least-once delivery with pending message recovery on restart.

Storage backends

ClickHouse

ClickHouse holds the three hot-path tables:

  • forge_siem.events — decoded events, 90-day TTL by default
  • forge_siem.alerts — alert records, 365-day TTL
  • forge_siem.raw_events — raw agent payloads, 90-day TTL

All tables use MergeTree with LowCardinality(String) for high-cardinality fields and Map(String,String) for dynamic extra fields. Bootstrap creates the database and tables idempotently on every deploy.

PostgreSQL

PostgreSQL holds operational state: agents, rules, CVEs, cases, hunts, response log, analyst accounts, notification destinations, and the audit log. Schema is applied by the bootstrap job via idempotent IF NOT EXISTS DDL in db/schema.sql.

ArgoCD setup

The Helm chart includes ArgoCD sync wave annotations. The bootstrap job runs at wave 1 (after wave 0 databases), services run at wave 2.

argocd — Application manifest
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: forge-siem
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/PlatformForgeTechnologies/forge-siem-platform
    targetRevision: main
    path: k8s-apps/siem
    helm:
      valueFiles: [values-pi.yaml]   # or your own overlay
  destination:
    server: https://kubernetes.default.svc
    namespace: siem
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true

Scaling with KEDA

When keda.enabled: true, KEDA ScaledObjects are created for decoder and rules-engine. They scale on Redis stream consumer lag — if the raw events stream falls behind, more decoder pods are spawned automatically.

Minimum replicas: 1. Maximum replicas: 10 (decoder), 5 (rules-engine). Adjust via values.yaml under decoder.keda.*.

All other services use standard HPA or static replica counts.

Upgrades

shell
helm repo update forge-siem
helm upgrade siem forge-siem/siem \
  --namespace siem \
  -f my-values.yaml

The bootstrap job re-runs on every upgrade via the PreSync hook, applying any new schema migrations idempotently. No manual DB migration steps are needed.

Note: If you are using ArgoCD with automated.selfHeal, upgrades happen automatically when the chart version changes in the repo. The bootstrap job will run before any service is updated.
